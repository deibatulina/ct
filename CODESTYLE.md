# CODESTYLE

Этот файл фиксирует стиль кода для проекта.

## Общий принцип

- Пиши код максимально просто.
- Выбирай самый понятный и прямой вариант реализации.
- Не усложняй архитектуру заранее.
- Не добавляй абстракции "на будущее", если они пока не нужны.
- Если задачу можно решить функцией и словарём, не вводи класс.

## Структура проекта

- `main.py` — точка запуска проекта.
- `config.py` — конфигурационные переменные и работа с `.env`.
- папка `src/` — основной код проекта, разбитый по логическим модулям.
- Не складывай основную бизнес-логику в `main.py`.
- `main.py` должен быть тонким: принять аргументы, загрузить конфиг, вызвать нужную функцию.

## Конфиги

- Конфиг должен быть максимально простым и интуитивным.
- Не используй классы для конфигов без явной необходимости.
- Не используй `dataclass` для конфигов.
- Предпочитай обычные словари и простые функции загрузки конфигурации.
- Значения из `.env` должны загружаться в `config.py`.
- Пути и переменные окружения должны быть централизованы в `config.py`.

## Простота кода

- Не используй сложные паттерны без необходимости.
- Не делай глубокую иерархию классов.
- Не выноси код в отдельную сущность, если это не делает проект заметно понятнее.
- Предпочитай маленькие функции с ясным назначением.
- Избегай "магии" и неочевидного поведения.

## Типизация

- Типизация не нужна.
- Не добавляй type hints в функции, переменные и возвращаемые значения.
- Не усложняй код типами ради красоты.

## Классы и dataclass

- Не используй классы без явной необходимости.
- Не используй `dataclass`, если можно обойтись словарём, списком и функциями.
- Если класс всё же действительно нужен, он должен решать реальную задачу, а не просто "делать код более объектным".

## Функции

- Функции должны быть короткими и понятными.
- Каждая функция должна делать одну понятную вещь.
- Старайся использовать понятные имена функций и переменных.
- Если функция сложная, добавляй docstring на русском языке.
- Если функция простая и очевидная, docstring не обязателен.

## Docstring

- Для сложных функций docstring обязателен.
- Docstring должен быть на русском языке.
- Docstring должен коротко отвечать на вопрос: что делает функция и зачем она нужна.
- Не пиши формальные и перегруженные docstring.

Пример:

```python
def preprocess_rows(rows):
    """
    Собирает сырые строки датасета в итоговый формат для обучения.
    Удаляет пустые записи, нормализует текст и подготавливает поля.
    """
```

## main.py

- В `main.py` не должно быть тяжёлой логики.
- В `main.py` не должно быть длинных функций обработки данных.
- `main.py` должен:
  - разобрать аргументы командной строки;
  - загрузить конфиг;
  - вызвать нужный модуль из `src/`;
  - вывести результат или ошибку.

## src/

- В `src/` хранится основная логика проекта.
- Код внутри `src/` должен быть разбит по смыслу.
- Разделяй код по модулям так, чтобы было легко понять, где:
  - загрузка данных;
  - предобработка;
  - обучение;
  - оценка;
  - утилиты.

## Стиль реализации

- Пиши код так, чтобы его было легко читать с первого раза.
- Предпочитай явные шаги вместо слишком компактных конструкций.
- Не делай код "умным" в ущерб читаемости.
- Лучше немного длиннее, но понятнее.
- Если есть выбор между "красиво" и "понятно", выбирай "понятно".

## Чего избегать

- лишних классов;
- лишней типизации;
- лишних dataclass;
- сложных абстракций;
- запутанных helper-слоёв;
- перегруженных универсальных функций;
- скрытой логики в `main.py`;
- конфигов, завязанных на классы и вложенные объектные структуры без необходимости.

## Ожидаемый стиль для LLM

- Сначала выбирай самый простой рабочий вариант.
- Не рефакторь в сложную архитектуру без запроса.
- Поддерживай плоскую и понятную структуру файлов.
- Используй словари и функции по умолчанию.
- Добавляй русские docstring только там, где логика действительно неочевидна.
- Любое усложнение должно быть легко объяснимо одной фразой: зачем оно нужно именно сейчас.
